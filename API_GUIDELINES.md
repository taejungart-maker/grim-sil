# 🎯 grim-sil 개발 가이드라인

## 🤖 클로드 13원칙 (작업 원칙)

### 1. 컨텍스트 관리
- 병렬 세션에서도 일관된 맥락 유지
- 대화 요약 및 핵심 정보 추적

### 2. 시스템 알림
- 사용자 입력이 필요할 때 명확히 알림
- 중요한 결정 사항은 사전 승인 요청

### 3. 고급 추론 모델 활용
- 복잡한 문제는 단계별 분석
- 여러 해결책 비교 후 최적안 제시

### 4. 권한 자동화
- 반복 명령은 자동 실행 제안
- `// turbo` 어노테이션으로 안전한 명령 자동 승인

### 5. 서브에이전트 위임
- 브라우저 작업, 파일 탐색 등 전문 도구 활용
- 효율적인 작업 분배

### 6. 실용적 접근
- 이론보다 실용성 우선
- 빠른 피드백 루프

### 7. 문서화
- 작업 내용 기록 유지
- 가이드라인 파일로 지식 보존

### 8. 점진적 개선
- 완벽보다 동작하는 코드 우선
- 반복적으로 개선

### 9. 사용자 중심
- 사용자 요청 최우선
- 불필요한 질문 최소화

### 10. 투명성
- 진행 상황 명확히 공유
- 오류 발생 시 솔직히 보고

### 11. 안전성
- 파괴적 작업은 반드시 확인
- 백업 및 롤백 고려

### 12. 효율성
- 병렬 처리 가능하면 동시 실행
- 불필요한 반복 제거

### 13. 지속성
- 세션 간 컨텍스트 유지
- 프로젝트 지식 파일로 보존

## 🔧 API 설계 원칙

### 1. 실용적 REST
- 이론에 과도하게 집착하지 않음
- 빠른 구현과 유지보수성 우선

### 2. HATEOAS (기본 적용)
- API 응답에 관련 액션 링크 포함
```json
{
  "id": "123",
  "title": "작품명",
  "_links": {
    "self": "/api/artworks/123",
    "edit": "/api/artworks/123/edit",
    "delete": "/api/artworks/123"
  }
}
```

### 3. Swagger 문서화
- 모든 API 엔드포인트 문서화
- 요청/응답 스키마 명세

### 4. DTO 설계
- DB 구조가 프론트에 노출되지 않도록
- 응답용 DTO와 요청용 DTO 분리
